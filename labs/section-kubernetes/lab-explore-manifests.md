# Lab: Validate the Kubernetes Manifests

## 🎯 Lab Goal

Verify that the manifests you wrote are correct and that Kustomize wires them together as expected, before deploying to a real cluster. This is a structured pre-flight checklist.

## 📝 Overview & Concepts

Writing YAML by hand introduces room for mistakes: wrong indentation, a mismatched label, a namespace assignment that was missed. Running `kubectl kustomize k8s/` renders the complete final YAML without touching the cluster, making it a safe way to catch problems early.

The key things to verify before deploying:

- Every resource is in the correct namespace
- Labels and selectors match between Deployments and Services
- The OTLP endpoint uses the cross-namespace FQDN
- Image tags were injected by Kustomize
- ConfigMap names in Deployments match the generated (hashed) ConfigMap names

## 📋 Tasks

**1. Render the full manifest set**

Run Kustomize and save the output to a file for easy inspection:

```bash
kubectl kustomize k8s/ > /tmp/k8s-rendered.yaml
```

Open `/tmp/k8s-rendered.yaml` and scroll through it. You should see all resources from both `k8s/app/` and `k8s/observability/` in a single stream.

**2. Verify namespaces**

Search the rendered output for `namespace:` fields:

```bash
grep "namespace:" /tmp/k8s-rendered.yaml | sort | uniq -c
```

Every resource (except `ClusterRole` and `ClusterRoleBinding`, which are cluster-scoped) should have either `namespace: app` or `namespace: observability`. If any resource is missing a namespace, check its parent `kustomization.yaml`.

**3. Verify the OTLP endpoint**

Confirm the cross-namespace DNS name is present in both the frontend and worker Deployments:

```bash
grep "OTEL_EXPORTER_OTLP_ENDPOINT" /tmp/k8s-rendered.yaml -A1
```

Both values should show `http://otel-collector.observability.svc.cluster.local:4318`. A short name like `otel-collector:4318` will cause silent telemetry loss at runtime.

**4. Verify image tags**

Confirm the Kustomize image transformer applied the version tag:

```bash
grep "lmacademy/" /tmp/k8s-rendered.yaml
```

Both lines should show the `:v1.3.0` tag. If you see `lmacademy/web-translator-frontend` without a tag, the `images:` block in `k8s/kustomization.yaml` is not matching the image name correctly.

**5. Verify ConfigMap names**

Confirm that ConfigMaps were generated with a hash suffix and that the Deployments reference the same hashed names:

```bash
# List generated ConfigMap names
grep -E "^  name: .*-[a-z0-9]{8,10}$" /tmp/k8s-rendered.yaml

# Confirm Deployments reference hashed names
grep "configMap:" /tmp/k8s-rendered.yaml -A2
```

The `configMap.name` in each Deployment's `volumes:` block should match one of the generated ConfigMap names. If a Deployment still references the plain name (e.g. `otel-collector-config` without a hash), Kustomize did not resolve the reference. Check that the `configMapGenerator` name in `kustomization.yaml` exactly matches the name used in the Deployment's `volumes:` block.

## 🤖 AI Checkpoints

1. **Cross-Namespace DNS:**

   Ask your AI assistant: "Why can't the worker use just `otel-collector:4318` to reach the OpenTelemetry Collector in Kubernetes, even though that worked in Docker Compose? What is the full DNS resolution path for `otel-collector.observability.svc.cluster.local`?"

   **What to evaluate:** Does it explain that Kubernetes DNS resolves short names only within the same namespace, so `otel-collector` would look for a Service named `otel-collector` in the `app` namespace and fail? Does it describe the FQDN structure as `<service>.<namespace>.svc.<cluster-domain>`, where `cluster.local` is the default cluster domain? Does it mention that this cross-namespace dependency is a common source of misconfiguration when migrating from Compose?

2. **Kustomize vs. Helm:**

   Ask: "This project uses Kustomize rather than Helm. What are the tradeoffs between the two? For a course project like this, why might Kustomize be a better choice than Helm?"

   **What to evaluate:** Does it explain that Helm uses Go templating to parameterise YAML (powerful but complex), while Kustomize patches plain YAML without templating (simpler, no new language)? Does it note that Kustomize is built into `kubectl` so there is nothing to install? Does it mention that for straightforward deployments with a small number of variations, Kustomize is easier to read and debug; the rendered output is just YAML, not a template?

3. **ConfigMap vs. Bind Mount:**

   Ask: "One of the observability services (e.g., Prometheus) uses a ConfigMap generated by Kustomize. If I update `k8s/observability/configs/prometheus.yml` and re-run `kubectl apply -k k8s/observability/`, will the running Prometheus pod pick up the change automatically? Why or why not?"

   **What to evaluate:** Does it explain that `kubectl apply` updates the ConfigMap resource, but pods mounting it as a volume may not reload automatically; it depends on whether the application watches the mounted file for changes? Does it mention that by default, Kubernetes syncs ConfigMap-mounted volumes after a short delay (kubelet sync interval), but application processes may need to be restarted to pick up the change? Does it suggest strategies like a rolling restart (`kubectl rollout restart`) or using a config-reloading sidecar for production use?
